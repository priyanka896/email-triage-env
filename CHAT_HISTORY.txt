3. Key findings from the framework:
   - Environment must extend Environment[ActT, ObsT, StateT]
   - Must implement: reset(), step(), state (property)
   - Models must extend Action, Observation, State (Pydantic BaseModel)
   - Observation has built-in fields: done (bool), reward (float|None), metadata (dict)
   - serialize_observation() strips done/reward from obs dict, puts them at top level
   - HTTP endpoints are stateless (new env per request)
   - WebSocket /ws endpoint is stateful (session-based)
   - openenv.yaml is the manifest file
   - create_app(EnvClass, ActionClass, ObsClass, env_name) creates the FastAPI app

PHASE 2: ENVIRONMENT DESIGN DECISIONS
================================================================================

Chose: EMAIL TRIAGE — a real-world task where agents classify, prioritize,
and draft replies for customer support emails.

Why this task:
- Genuinely real-world (support teams do this daily)
- Not a game or toy
- Natural difficulty scaling (clear emails → ambiguous → complex)
- Rich action space (priority + category + reply text + escalation flag)
- Supports partial credit scoring across multiple dimensions

Task structure:
- 3 tasks × 3 emails each = 9 total emails
- easy_triage:   Clear signals (login issue, feature request, billing question)
- medium_triage: Ambiguous (performance bug, partnership inquiry, follow-up invoice)
- hard_triage:   Complex (production data loss + legal threat, spam detection,
                 mixed bug+feature thread with history)

Grading dimensions (per email, total 1.0):
- Priority match:   0.30 (exact=0.30, adjacent=0.15, else 0)
- Category match:   0.30 (exact=0.30, else 0)
- Escalation match: 0.15 (correct=0.15, else 0)
- Reply quality:    0.25 (keyword coverage ratio)
- Penalty: replies < 3 words lose 0.1

PHASE 5: KEY TECHNICAL DETAILS
================================================================================

Serialization format (from OpenEnv framework):
  Response = {"data": {"observation": {...}, "reward": float, "done": bool}}
  The "done" and "reward" fields are EXCLUDED from the observation dict
  and placed at the envelope level by serialize_observation().

WebSocket protocol:
  Reset:  {"type": "reset", "data": {"task_id": "easy_triage"}}
  Step:   {"type": "step", "data": {"priority": "...", "category": "...", ...}}
  State:  {"type": "state"}  (no data field)

Inference script:
  - Uses OpenAI client with API_BASE_URL, MODEL_NAME, HF_TOKEN env vars
  - Connects via WebSocket for stateful sessions
  - System prompt instructs LLM to output JSON with priority/category/reply/escalate
  - Parses JSON from LLM response (handles markdown fences)
  - Sanitizes action fields to valid enum values
  - Runs all 3 tasks sequentially, prints per-task and average scores
